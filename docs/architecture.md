# Архитектура и план MVP

> Серьёзная масштабируемая платформа. Приоритет — корректность денег, затем способность расти.

## 1. Стек

| Слой | Выбор | Почему |
|---|---|---|
| Backend | Go, **модульный монолит** | Корректность денег, перформанс, конкурентность (много воркеров-вотчеров), один бинарь |
| БД | PostgreSQL | Транзакционность леджера. Деньги = `bigint` наносы (1 TON = 1e9) |
| Кэш / локи | Redis | Кэш коэффициентов, rate-limit, распределённые локи |
| Очередь задач | River (Postgres-backed) | Транзакционно с БД, минимум инфры; позже → NATS/Kafka |
| Frontend | Telegram Mini App: React + TS (Vite) | UI рынков, TON Connect |
| Бот | Go (telebot), тонкий | Подключён к Telegram **Business-аккаунту** для подарков |
| TON | tonutils-go + 2 RPC-провайдера (failover) | Депозит-вотчер, вывод, hot/cold-кошельки |
| Подарки | Сервис-оракул цен: Portals / Tonnel / MRKT | Флор уникальных; Stars→TON для обычных |
| Инфра | Docker, Postgres PITR, Prometheus+Grafana, OTel, Sentry | Наблюдаемость и DR с первого дня |

**Почему монолит, а не микросервисы:** ставка + движение денег обязаны быть в одной DB-транзакции. Монолит с жёсткими границами модулей даёт это бесплатно; фоновые воркеры уже отдельные процессы — их и отщепляем в сервисы первыми по мере роста.

## 3. Деньги и леджер (сердце системы)

**Инвариант:** деньги двигаются **только** через сбалансированную леджер-транзакцию (сумма проводок = 0). Баланс никогда не меняется в обход. Проверяется DB-триггером + ночным job'ом целостности (`sum(entries)=0`, `balance == Σentries`).

Типы счетов (`accounts`): `USER_BALANCE`, `BET_ESCROW`, `LIABILITY_RESERVE`, `HOUSE_TREASURY`, `FEE_REVENUE`, `GIFT_INVENTORY`, `EXTERNAL_TON`, `EXTERNAL_GIFT`.

Ключевые таблицы: `users`, `accounts`, `ledger_transactions`, `ledger_entries`, `account_balances` (материализованный баланс), `markets`, `outcomes`, `odds_history`, `bets`, `risk_exposure`, `deposits`, `withdrawals`, `gifts_inventory`, `gift_price_feed`, `ton_deposit_intents`, `idempotency_keys`, `audit_log`, `outbox`.

### Модель ставки (структурная платёжеспособность дома)

Дом не может принять ставку, которую не сможет выплатить:

```
Ставка (S — ставка, P = S × odds — выплата):
  USER_BALANCE      −S    →  BET_ESCROW        +S       (заморозили ставку игрока)
  HOUSE_TREASURY  −(P−S)  →  LIABILITY_RESERVE +(P−S)   (зарезервировали профит дома)
  ↳ если в TREASURY не хватает на резерв → ставка отклоняется (книга защищена)

Выигрыш:
  BET_ESCROW        −S    →  USER_BALANCE      +S        (вернули ставку)
  LIABILITY_RESERVE −(P−S)→  USER_BALANCE   +(P−S)       (выплатили профит)

Проигрыш:
  BET_ESCROW        −S    →  HOUSE_TREASURY    +S        (дом забрал ставку)
  LIABILITY_RESERVE −(P−S)→  HOUSE_TREASURY +(P−S)       (вернули резерв)
```

Каждый открытый риск уже обеспечен резервом в леджере → книга не может стать неплатёжеспособной.

## 4. Ключевые потоки

- **Депозит TON.** Deposit-intent (единый hot-кошелёк + уникальный comment на юзера) → вотчер чейна → при финализации матчим comment → `EXTERNAL_TON → USER_BALANCE`. Идемпотентность по tx-hash. Нет comment → ручная очередь.
- **Депозит подарком.** Подарок на Business-аккаунт → оценка (уникальный = TWAP-флор × (1−haircut); обычный = Stars→TON × haircut) → `EXTERNAL_GIFT → USER_BALANCE` + подарок в инвентарь. ⚠️ TWAP, не спот; лимит кредита на подарок и velocity-лимит на юзера (анти-накрутка флора).
- **Ставка (hot path).** Валидация (рынок OPEN, лимиты, баланс) → сверка коэффициента (drift > tolerance → отклонить с новым) → risk-check (лимит экспозиции на исход) → в одной DB-tx: лок строки баланса, проводки по модели выше, запись `bet` с зафиксированным odds, апдейт `risk_exposure`, idempotency-key.
- **Сеттлмент.** Рынок после close_time → `RESOLVING` → опрос резолва Polymarket (UMA). **Не автосеттлим при споре** — задержка + admin override. По каждой ставке идемпотентно. `CANCELLED` → возврат ставок.
- **Вывод TON.** Валидация + анти-фрод (кулдаун после депозита, дневные лимиты, KYC-порог) → `USER_BALANCE → WITHDRAWAL_PENDING` + outbox → процессор шлёт из hot-кошелька (сериализация по seqno против двойной отправки) → confirm → `→ EXTERNAL_TON`; fail → возврат. Крупные суммы → cold/multisig + ручной апрув. Ключи — envelope encryption.
- **Вывод подарком.** Каталог (инвентарь + флор) → списание → выдача: из инвентаря `transferGift` (учитывая кулдаун трансфера Telegram) либо покупка на маркете → отправка. Fail → возврат.

## 5. Коэффициенты и риск

**Odds-движок:** implied prob `p` из Polymarket → `odds = 1 / p_adj`, где `p_adj` включает overround (маржа, напр. 5%, шире на волатильных категориях). **Динамический шейдинг:** при высокой экспозиции на исход режем его odds, поднимаем встречный — балансируем книгу без хеджа. Odds **фиксируется при ставке**.

**Risk-движок:** лимит ответственности на исход / рынок / юзера, max-bet по ликвидности, глобальный дневной risk-budget, liability-reserve в леджере, kill-switch на рынок и глобальный, дашборд экспозиции + алерт `treasury < liability`. v2 — реальный хедж встречной позицией на самом Polymarket.

## 6. Анти-абьюз

- **Арбитраж депозит↔вывод подарков** (главный риск): консервативная оценка (TWAP + haircut), асимметричный спред (кредит < стоимости вывода), кулдаун вывода после депозита, кап на подарочные депозиты.
- **Снайпинг устаревших odds:** серверная пересверка + suspend на быстрых рынках.
- **AML** (реальный вывод): депозит→вывод без ставок блокируем (кулдаун, min-оборот, KYC-порог, sanctions-screening адресов).
- **Мультиаккаунты:** сигналы device/IP/возраст аккаунта, лимиты новичкам.

## 7. Масштаб и надёжность

Stateless API за LB (горизонталь) · Postgres primary + read-replicas, партиционирование `ledger_entries`/`bets`/`odds_history` по времени, шардинг по `user_id` при упоре · все воркеры идемпотентны + транзакционный outbox · TON — мультипровайдер, пороги подтверждений, reorg, seqno-сериализация вывода · **приоритет корректности над доступностью** — при сомнении отклоняем ставку/паузим вывод · ночной job целостности леджера.

## 8. Trade-offs

| Решение | Взято | Цена / когда пересмотреть |
|---|---|---|
| Монолит vs микросервисы | Монолит | Дисциплина границ; отщеплять воркеры по росту |
| Дом (букмекер) vs CLOB/AMM | Дом | Нужен капитал + risk-движок; P2P/AMM в v2 |
| Зависимость от Polymarket | Единственный источник | Вендор-риск, лаг резолва → интерфейс `MarketSource`, admin override |
| Postgres-очередь vs Kafka | River | Потолок throughput → NATS/Kafka на объёмах |
| Кастоди | Self hot/cold + multisig | Бремя безопасности; альтернатива — кастоди-провайдер |
| Депозит по comment vs subaddress | comment | Забытый comment → ручная сверка |

## 9. План MVP по фазам

Порядок: сначала корректные денежные рельсы → ставочный цикл в песочнице → источник данных → реальный ввод/вывод (TON проще подарков) → подарки (самое хрупкое) → масштаб. Каждая фаза снимает риск следующей.

- **Фаза 0 — Фундамент.** Скелет монолита, Postgres, миграции, CI. Леджер (двойная запись, наносы, инвариант, audit_log, идемпотентность, job целостности). Auth (Mini App initData). Наблюдаемость.
- **Фаза 1 — Ставочный цикл (закрытая экономика).** Рынки/исходы (создание админом), odds v0, betting-движок (escrow + liability-reserve), risk-капы, сеттлмент (ручной). Mini App: список рынков, ставка, мои ставки, баланс. *Тест на «игровом» балансе.*
- **Фаза 2 — Polymarket.** Ingester (события+исходы+implied prob), odds v1 (авто-маржа), авто-резолв с задержкой, dispute-guard, admin override.
- **Фаза 3 — TON-рельсы (реальная ценность).** Депозит-вотчер, withdrawal-процессор (hot/cold, outbox, seqno), управление ключами, анти-фрод/AML, кулдауны/лимиты.
- **Фаза 4 — Подарки.** Business-аккаунт, приём+оценка (оракул TWAP+haircut), инвентарь, вывод подарком (трансфер + покупка на маркете, кулдауны).
- **Фаза 5 — Hardening/scale.** Risk-дашборд, динамический шейдинг, kill-switch, реплики/партиции, нагрузочный тест, DR-учения, responsible gambling + гео-фенс.
- **v2:** P2P/AMM-ликвидность, хедж на Polymarket, лидерборды/копитрейдинг ставок, турниры.
