# gifts-userbot

Сервис-покупатель коллекционных подарков для подарочного вывода (путь B). Это
**user-аккаунт** под управлением MTProto (Pyrogram), а **НЕ бот** — Bot API не умеет
покупать коллекционные подарки за TON. Покупает подарок на Portals и передаёт его
получателю через `payments.transferStarGift`. Наружу — внутренний HTTP для Go-бэка.

## ⚠️ Аккаунт — отдельный, не личный

Автоматизация user-аккаунта — серая зона ToS (риск бана), а session-string = полный
доступ к аккаунту. Поэтому используем **отдельный расходный аккаунт** на отдельный
номер, не личный Артёма.

## Чек-лист настройки (делает Артём)

1. **Отдельный Telegram-аккаунт** на отдельный номер (виртуальный/вторая SIM).
2. **api_id / api_hash** — залогинься этим аккаунтом на https://my.telegram.org →
   *API development tools* → создай приложение.
3. **Session-string:** `pip install -r requirements.txt` → `python gen_session.py`
   (запускать ЛОКАЛЬНО, введёт номер+код). Готовую строку — в `.env` как
   `USERBOT_SESSION`. ⚠️ Секрет, в чат не вставлять.
4. **TON-кошелёк юзербота** — заведи новый, зафандь небольшим количеством TON (это
   флоат под выкуп подарков). Мнемонику — в `.env` как `USERBOT_TON_MNEMONIC`.
5. **Маржа** — `GIFT_MARGIN` в `.env` (по умолчанию 0.15 = +15%).
6. Скопируй `.env.example` → `.env`, заполни, положи на сервер (не коммить).

## Что дальше (делаю я, после твоей настройки)

**Де-риск-прототип** (до полной интеграции) — проверить главные неизвестные:
- авторизация Portals (initData / Pyrogram-логин, срок жизни токена);
- получить листинги подарков с ценами в TON;
- тестовая покупка + передача на расходном аккаунте;
- **замер кулдауна передачи** (`next_transfer_date`): можно ли отдать сразу или есть
  задержка → решает UX (мгновенная доставка vs очередь).

Потом: Go-эндпоинты (каталог + атомарная покупка с дебетом TON и возвратом при
фейле), сетка подарков на фронте вместо «скоро», контейнеризация в docker-compose.

См. память: `predict-withdrawals.md`.
